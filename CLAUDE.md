# 大規模改修・デザイン変更のルール

## 必須：Geminiとの事前壁打ち

以下のいずれかに該当する作業を行う前に、
必ずGemini CLIで設計を壁打ちしてから実装に入ること。

### 壁打ちが必要なケース
- UIの全面・部分的なデザイン変更
- 新機能の追加（画面・モーダル・フォームの新設）
- 既存のロジックの大きな変更（計算式・データ構造）
- 複数ファイルにまたがる改修
- Firestoreのコレクション構造変更

### 壁打ちの手順
1. ターミナルで以下を実行してGeminiに相談する
   gemini

2. 以下の内容をGeminiに伝えて意見をもらう
   - やりたいこと（要件）
   - 現在の実装（該当ファイルの内容）
   - 懸念点・考えているアプローチ

3. Geminiの回答を踏まえて実装方針を固める

4. 方針が固まったらClaudeで実装する

### 壁打ちのプロンプト例
「RegalCastというスタッフ管理アプリを開発中です。
[やりたいこと] を実装したいと思っています。
現在の実装は [現状] です。
[アプローチA] と [アプローチB] で迷っています。
UX・保守性・パフォーマンスの観点でどちらが良いですか？」

## 実装前チェックリスト
- [ ] Geminiと壁打ち済み
- [ ] 実装方針が決定している
- [ ] 既存機能への影響範囲を把握している
- [ ] Firestoreデータへの影響がないことを確認した
- [ ] モバイルファーストで設計されている
